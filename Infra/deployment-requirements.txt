============================================================
  MediAssist-AI — Post-Deployment Manual Steps
  (required after running deploy-all.sh for the first time)
============================================================

After deploy-all.sh completes, the Azure infrastructure and the initial
code deployment are fully automated. The steps below require a one-time
manual action and cannot be automated without additional credentials.

─────────────────────────────────────────────────────────────
STEP 1 — Verify the Initial Deployment
─────────────────────────────────────────────────────────────
1.1  Open the App Service URL printed at the end of deploy-all.sh
     (e.g. https://webapp-mediassist-<suffix>.azurewebsites.net)

1.2  The React frontend should load in the browser.
     If you see an empty page, check App Service logs:
       az webapp log tail --name <WEBAPP_NAME> --resource-group <RG>

1.3  Verify the Python agent bridge is running:
       curl https://<webapp-url>/api/health
       curl https://<webapp-url>/api/agent/health  (or equivalent)

─────────────────────────────────────────────────────────────
STEP 2 — Set Up GitHub Actions CI/CD (OIDC Authentication)
─────────────────────────────────────────────────────────────
GitHub Actions uses OIDC (OpenID Connect) to authenticate to Azure.
Only 4 bootstrap identifiers need to be stored as GitHub secrets.
All other config (webapp name, resource group, VITE URLs) is fetched
from Key Vault at runtime by the 'resolve-config' workflow job.

deploy-all.sh attempts to set the 4 secrets automatically using the
GitHub CLI (gh). If it printed "GitHub CLI not found" or any secret
shows "SET_MANUALLY", follow these steps:

2.1  Install the GitHub CLI and authenticate:
       brew install gh          # macOS
       gh auth login

2.2  Set the 4 bootstrap secrets (all values in .deployment-summary.txt):
       gh secret set AZURE_CLIENT_ID       --body "<appRegistrationClientId>"
       gh secret set AZURE_TENANT_ID       --body "<tenantId>"
       gh secret set AZURE_SUBSCRIPTION_ID --body "<subscriptionId>"
       gh secret set AZURE_KEY_VAULT_NAME  --body "<kvName>"

     That's it. No webapp name, resource group, or URLs in GitHub.
     The workflow reads WEBAPP-NAME and RESOURCE-GROUP-NAME from
     Key Vault after logging in with OIDC.

2.3  Add a Federated Credential for OIDC login.
     The App Service Managed Identity is a Service Principal, not an
     App Registration, so OIDC federated credentials are not available
     on it directly. You have TWO options:

     OPTION A (Recommended — Service Principal with federated credential):
       a) Create an App Registration in Azure Entra:
            az ad app create --display-name "mediassist-github-actions"
            az ad sp create --id <appId>
       b) Add federated credential for your GitHub repo:
            az ad app federated-credential create \
              --id <appId> \
              --parameters '{
                "name": "mediassist-github-main",
                "issuer": "https://token.actions.githubusercontent.com",
                "subject": "repo:<org>/<repo>:ref:refs/heads/main",
                "audiences": ["api://AzureADApplication"]
              }'
            # Repeat for the "dev" branch:
            az ad app federated-credential create \
              --id <appId> \
              --parameters '{
                "name": "mediassist-github-dev",
                "issuer": "https://token.actions.githubusercontent.com",
                "subject": "repo:<org>/<repo>:ref:refs/heads/dev",
                "audiences": ["api://AzureADApplication"]
              }'
       c) Grant Contributor role to the service principal on the resource group:
            az role assignment create \
              --assignee <appId> \
              --role Contributor \
              --scope /subscriptions/<subscriptionId>/resourceGroups/<resourceGroup>
       d) Update the AZURE_CLIENT_ID secret to use the App Registration client ID:
            gh secret set AZURE_CLIENT_ID --body "<appRegistrationClientId>"

     OPTION B (Quick — use service principal client secret instead of OIDC):
       a) Create a service principal with a secret:
            az ad sp create-for-rbac \
              --name "mediassist-github-actions" \
              --role Contributor \
              --scopes /subscriptions/<subscriptionId>/resourceGroups/<resourceGroup> \
              --sdk-auth
       b) Store the JSON output as a single secret:
            gh secret set AZURE_CREDENTIALS --body '<json output>'
       c) Change the login step in .github/workflows/main.yml from:
              uses: azure/login@v2
              with:
                client-id: ${{ secrets.AZURE_CLIENT_ID }}
                tenant-id: ${{ secrets.AZURE_TENANT_ID }}
                subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
          to:
              uses: azure/login@v1
              with:
                creds: ${{ secrets.AZURE_CREDENTIALS }}

─────────────────────────────────────────────────────────────
STEP 3 — Upload ICD-10 Knowledge Base Files to Blob Storage
─────────────────────────────────────────────────────────────
The RAG pipeline requires two knowledge base files in Azure Blob Storage.
These are not committed to the repository (file size / licensing).

3.1  Obtain the files:
       - icd_10_cm_october_2025_guidelines_0.pdf   (ICD-10-CM PDF)
       - section111_valid_icd10_october2025.csv    (ICD-10 valid codes CSV)
     Source: https://www.cms.gov/medicare/coding-billing/icd-10-codes

3.2  Get the storage connection string from your local .env or Key Vault:
       STORAGE_CONN=$(az keyvault secret show \
         --vault-name <kvName> \
         --name AZURE-STORAGE-CONNECTION-STRING \
         --query value -o tsv)

3.3  Upload the files:
       az storage blob upload \
         --account-name <storageAccountName> \
         --container-name az-medassistai-container1 \
         --file ./icd_10_cm_october_2025_guidelines_0.pdf \
         --name  icd_10_cm_october_2025_guidelines_0.pdf \
         --connection-string "$STORAGE_CONN"

       az storage blob upload \
         --account-name <storageAccountName> \
         --container-name az-medassistai-container2 \
         --file ./section111_valid_icd10_october2025.csv \
         --name  section111_valid_icd10_october2025.csv \
         --connection-string "$STORAGE_CONN"

STEP 4 — Run the AI Search Embedding Pipeline (Optionally - Create Indexers in AI Search Service)
─────────────────────────────────────────────────────────────
The AI Search indexes are created by deploy-all.sh but are empty.
You must run the embedding pipeline to populate them.

4.1  Set up local Python environment:
       cd Backend/summarizer_agent
       python3 -m venv .venv && source .venv/bin/activate
       pip install -r requirements.txt

4.2  Copy .env from deploy-all.sh output (already written to this dir).

4.3  Run the knowledge base preparation script:
       python prepare_kb.py

     This will:
     - Download the ICD-10 PDF from Blob Storage
     - Chunk and embed the content using Azure OpenAI text-embedding-3-small
     - Upload vectors to az-med-cont1-index-final (primary index)

4.4  Optionally run the RAG pipeline test to verify:
       python e2e_test_runner.py

─────────────────────────────────────────────────────────────
STEP 6 — (Optional) Add Custom Domain / SSL
─────────────────────────────────────────────────────────────
The App Service includes a free *.azurewebsites.net HTTPS certificate.
To add a custom domain:
  az webapp config hostname add \
    --webapp-name <webappName> \
    --resource-group <resourceGroup> \
    --hostname <yourdomain.com>

Then bind a managed certificate:
  az webapp config ssl bind \
    --name <webappName> \
    --resource-group <resourceGroup> \
    --certificate-type managed \
    --hostname <yourdomain.com>

─────────────────────────────────────────────────────────────
STEP 7 — (Production) Rotate Auto-Generated Secrets
─────────────────────────────────────────────────────────────
deploy-all.sh auto-generates the following secrets and stores them
in Azure Key Vault. For production, replace them with your own:

  JWT_SECRET                — JSON Web Token signing key
  INTERNAL_API_SECRET       — Internal service-to-service authentication
  COSMOS_DB_CONNECTION_STRING — MongoDB vCore password

To update a secret:
  az keyvault secret set \
    --vault-name <kvName> \
    --name <SECRET-NAME> \
    --value "<new-secret-value>"

The App Service reads secrets via Key Vault references — no restart needed.

─────────────────────────────────────────────────────────────
STEP 8 — Run Agents Locally (After Deployment)
─────────────────────────────────────────────────────────────
All secrets live ONLY in Azure Key Vault. To run any agent locally,
DefaultAzureCredential needs to be able to authenticate and read from KV.
You need a service principal (SP) with "Key Vault Secrets User" role.

8.1  Create a service principal for local dev (one time):
       az ad sp create-for-rbac \
         --name "mediassist-local-dev" \
         --role "Key Vault Secrets User" \
         --scopes /subscriptions/<subscriptionId>/resourceGroups/<resourceGroup>/providers/Microsoft.KeyVault/vaults/<kvName>

     Save the JSON output — you'll need appId, password, and tenant.

8.2  Fill in the REPLACE_ values in each agent's .env file:

     File: deploy/.env  (Node.js backend)
     File: Backend/orchestrator_agent/.env
     File: Backend/summarizer_agent/.env
     File: Backend/Publishing_Agent/.env
     File: Backend/conversational_agent/.env

     Values to replace in each file:
       AZURE_KEY_VAULT_URI   → from .deployment-summary.txt
                               e.g. https://kv-medai-XXXXXXXX.vault.azure.net/
       AZURE_TENANT_ID       → from JSON output above (field: "tenant")
       AZURE_CLIENT_ID       → from JSON output above (field: "appId")
       AZURE_CLIENT_SECRET   → from JSON output above (field: "password")

     Also replace REPLACE_ prefixes on endpoint URLs:
       AZURE_OPENAI_ENDPOINT → from .deployment-summary.txt
       AZURE_SEARCH_ENDPOINT → from .deployment-summary.txt

8.3  Verify Key Vault access:
       az keyvault secret list --vault-name <kvName> --query "[].name" -o tsv

8.4  Start agents locally:
       # Node.js backend
       cd deploy && npm install && node src/app.js

       # Python orchestrator
       cd Backend/orchestrator_agent && pip install -r requirements.txt
       uvicorn api_server:app --host 0.0.0.0 --port 8000

       # Python summarizer (if running standalone)
       cd Backend/summarizer_agent && pip install -r requirements.txt
       python main.py
